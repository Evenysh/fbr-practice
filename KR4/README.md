# Контрольная работа №4
## Практические работы 19–23

В рамках контрольной работы №4 последовательно отрабатывались подходы к **серверной разработке и инфраструктуре**: работа с **реляционной** и **документоориентированной** СУБД, построение **REST API** и защищённого приложения с **ролями и кэшированием**, а также **балансировка нагрузки** и **отказоустойчивость** с помощью **Nginx**, **HAProxy** и **Docker**.

Контрольная работа №4 объединяет результаты практических заданий **19–23** и сопровождается данным `README.md`.

---

## Используемые технологии

В рамках выполнения практических работ использовались: 

- **Node.js**
- **Express**
- **PostgreSQL**, драйвер **pg**
- **MongoDB**, **Mongoose**
- **Redis** (кэш)
- **JSON Web Token (jsonwebtoken)** — access- и refresh-токены
- **bcryptjs** — хеширование паролей
- **multer** — загрузка файлов
- **CORS**
- **React**, **Vite**, **React Router**, **Axios**
- **Docker**, **Docker Compose**
- **Nginx** (reverse proxy, upstream, `max_fails` / `fail_timeout`, `proxy_next_upstream`)
- **HAProxy** (round robin, HTTP health check)

## Практическая работа 19

На данном этапе реализован **backend на Express** с подключением к **PostgreSQL** и REST API для сущности «пользователь».

### Выполненные доработки

- настроен пул подключений `pg` к базе данных;
- реализована проверка соединения с БД (корневой маршрут с `SELECT NOW()`);
- реализован **CRUD** по маршрутам `/api/users`:
  - `POST` — создание пользователя (`first_name`, `last_name`, `age`, метки времени);
  - `GET` — список всех пользователей;
  - `GET /:id` — пользователь по идентификатору;
  - `PATCH /:id` — частичное обновление полей;
  - `DELETE /:id` — удаление записи;
- использованы **параметризованные запросы** для защиты от SQL-инъекций;
- обработаны типичные ошибки (валидация входных данных, `404`, `500`).

### Результат

В результате практики №19 получен сервис, демонстрирующий типичный **серверный слой поверх PostgreSQL**: постоянное хранение данных, REST-соглашения и работа с транзакционной СУБД.

**Каталог:** [`fbr-practice_19`](./fbr-practice_19).

## Практическая работа 20

На данном этапе реализован аналогичный по смыслу **REST API для пользователей**, но с хранением в **MongoDB** и описанием схемы через **Mongoose**.

### Выполненные доработки

- установлено соединение с MongoDB (`mongodb://127.0.0.1:27017/...`);
- описана **схема пользователя**: числовой `id`, ФИО, возраст, `created_at` / `updated_at`;
- при создании записи **автоинкремент** числового `id` на основе последнего документа;
- реализованы маршруты:
  - `POST /api/users` — создание;
  - `GET /api/users` — список с сортировкой;
  - `GET /api/users/:id` — поиск по числовому `id`;
  - `PATCH /api/users/:id` — частичное обновление;
  - `DELETE /api/users/:id` — удаление;
- корневой маршрут подтверждает, что API запущено.

### Результат

В результате практики №20 закреплены отличия **документной модели** от реляционной при той же внешней форме API: схемы Mongoose, работа с документами и жизненным циклом `save` / `findOneAndDelete`.

**Каталог:** [`fbr-practice_20`](./fbr-practice_20).

## Практическая работа 21

На данном этапе собрано **многослойное приложение**: backend с **аутентификацией, авторизацией по ролям, кэшем** и **загрузкой файлов**, плюс **клиент на React (Vite)**.

### Выполненные доработки (backend)

- **Регистрация и вход** (`/api/auth/register`, `/api/auth/login`), выдача **access**- и **refresh**-токенов, эндпоинт **обновления** токена и **`/api/auth/me`**;
- разделение ролей: **`admin`**, **`seller`**, **`user`** — middleware проверки JWT и роли;
- сценарии для **пользователей** (для администратора): список, карточка, редактирование, блокировка (`delete` как блокировка с инвалидацией кэша);
- сценарии для **товаров**: создание с **загрузкой изображения** (multer), список и детальная карточка, обновление, удаление (с учётом ролей);
- **Redis**: кэш ответов для списков и сущностей с TTL, инвалидация при изменениях; при недоступности Redis приложение продолжает работу без кэша;
- раздача статики загрузок через `/uploads`.

### Выполненные доработки (frontend)

- приложение на **React** с **React Router**;
- страницы входа и регистрации, каталог товаров, детали товара, админские сценарии для пользователей и товаров (в т.ч. редактирование);
- HTTP-клиент **Axios** с настройкой под API.

### Результат

В результате практики №21 получено целостное **fullstack**-решение: безопасный доступ по токенам, разграничение прав, кэширование на стороне сервера и полноценный UI для основных сценариев.

**Каталоги:** [`fbr-practice_21/backend`](./fbr-practice_21/backend), [`fbr-practice_21/frontend`](./fbr-practice_21/frontend).

## Практическая работа 22

На данном этапе развернута **контейнеризированная** схема с **несколькими экземплярами backend** и двумя вариантами **балансировки нагрузки**: **Nginx** и **HAProxy**.

### Выполненные доработки

- не менее двух рабочих backend на разных портах внутри сети Docker; дополнительный инстанс в роли **backup** для сценария с Nginx;
- **Nginx**: `upstream` с **round-robin**, директивы **`max_fails`** и **`fail_timeout`**, **`proxy_next_upstream`** для переключения при ошибках;
- **HAProxy**: балансировка round robin, **HTTP health check** (`/health`);
- **docker-compose** с **профилями** `nginx` и `haproxy` для выбора стека без конфликта портов (**8080** — Nginx, **8081** — HAProxy).

### Результат

В результате практики №22 продемонстрированы **распределение запросов**, поведение при отказе узла и практическая настройка **reverse proxy** уровня продакшена.

Подробные команды запуска, тесты циклом `curl` и пояснения по портам — в отдельном файле [`fbr-practice_22/README.md`](./fbr-practice_22/README.md).

**Каталог:** [`fbr-practice_22`](./fbr-practice_22).

## Практическая работа 23

На данном этапе реализован **компактный пример** балансировки: два контейнера backend за одним **Nginx** в **Docker Compose**.

### Выполненные доработки

- два сервиса backend с разными переменными окружения (`SERVER_ID`), одинаковый образ из `Dockerfile`;
- **Nginx** слушает порт **80**, проксирует на upstream из двух узлов;
- в конфигурации заданы **`max_fails` / `fail_timeout`**, таймауты прокси, заголовки `X-Real-IP` / `X-Forwarded-For`, **`proxy_next_upstream`** при ошибках и таймаутах.

### Результат

В результате практики №23 закреплена базовая схема **«один вход — несколько приложений»** в Docker без усложнения профилями HAProxy, удобная для быстрой демонстрации round-robin.

**Каталог:** [`fbr-practice_23`](./fbr-practice_23).

## Запуск проекта

### Практики 19–20

1. Запустить локально **PostgreSQL** (практика 19) или **MongoDB** (практика 20), создать БД и при необходимости таблицу `users` под ожидаемую схему.
2. В файле `server.js` соответствующей папки задать **параметры подключения** (хост, порт, имя БД, пользователь, пароль) под вашу среду.
3. Установить зависимости и запустить сервер:

```bash
cd fbr-practice_19   # или fbr-practice_20
npm install
node server.js
```

Сервер слушает **`http://localhost:3000`** (если порт не менялся).

### Практика 21

1. Запустить **Redis** (например, на `127.0.0.1:6379`). Без Redis сервер стартует, но кэш недоступен.
2. Backend:

```bash
cd fbr-practice_21/backend
npm install
node app.js
```

3. Frontend (отдельный терминал):

```bash
cd fbr-practice_21/frontend
npm install
npm run dev
```

Адрес фронтенда смотрите в выводе Vite (обычно `http://localhost:5173`); в `axios`/прокси при необходимости укажите URL API (`http://localhost:3000`).

### Практика 22

Из каталога `fbr-practice_22`:

**Nginx-балансировщик:**

```bash
docker compose --profile nginx up --build
```

Балансировщик: **`http://localhost:8080/`**.

**HAProxy:**

```bash
docker compose --profile haproxy up --build
```

Балансировщик: **`http://localhost:8081/`**.

Остановка: `docker compose down`. Дополнительные тесты и сценарии отказа узла — в [`fbr-practice_22/README.md`](./fbr-practice_22/README.md).

### Практика 23

Из каталога `fbr-practice_23`:

```bash
docker compose up --build
curl http://localhost/
docker compose stop backend1
```

Точка входа: **`http://localhost/`** (порт **80** на хосте).
