# Практические работы 7–11

В рамках практических работ 7–11 было последовательно разработано клиент-серверное веб-приложение с использованием **Node.js** для серверной части и **React.js** для клиентской части.

На каждом следующем этапе приложение дорабатывалось и расширялось: от базового REST API с регистрацией и товарами до полноценной системы с авторизацией, refresh-токенами, пользовательским интерфейсом и разграничением прав доступа по ролям.

---

## Практическая работа 7

На данном этапе было реализовано серверное приложение на **Node.js**, содержащее базовый набор маршрутов для работы с пользователями и товарами.

### Реализованные маршруты

| Маршрут | Метод | Описание |
|---|---|---|
| `/api/auth/register` | `POST` | Регистрация пользователя |
| `/api/auth/login` | `POST` | Вход в систему |
| `/api/products` | `POST` | Создание товара |
| `/api/products` | `GET` | Получение списка товаров |
| `/api/products/:id` | `GET` | Получение товара по идентификатору |
| `/api/products/:id` | `PUT` | Обновление параметров товара |
| `/api/products/:id` | `DELETE` | Удаление товара |

### Реализованные сущности

#### Пользователь

Для сущности **«Пользователь»** были реализованы следующие поля:

- `id`
- `email`
- `first_name`
- `last_name`
- `password`

В качестве логина используется поле `email`.  
Пароль пользователя хранится не в открытом виде, а в виде хеша, что обеспечивает более безопасную обработку данных.

#### Товар

Для сущности **«Товар»** были реализованы следующие поля:

- `id`
- `title`
- `category`
- `description`
- `price`

### Результат

В результате выполнения практической работы №7 было создано базовое серверное API, обеспечивающее регистрацию и вход пользователей, а также выполнение основных операций CRUD для товаров.

---

## Практическая работа 8

На данном этапе было выполнено расширение серверного приложения, созданного в практической работе №7. Основной задачей стало добавление механизма авторизации по токену и реализация защищённых маршрутов.

### Основные доработки

- реализована выдача `accessToken` при успешном входе пользователя в систему;
- добавлен защищённый маршрут:

| Маршрут | Метод | Описание |
|---|---|---|
| `/api/auth/me` | `GET` | Получение объекта текущего пользователя |

Данный маршрут возвращает информацию о текущем авторизованном пользователе на основе переданного токена.

### Защищённые маршруты

Также были защищены следующие маршруты для работы с товарами:

- `GET /api/products/:id`
- `PUT /api/products/:id`
- `DELETE /api/products/:id`

### Результат

В результате выполнения практической работы №8 приложение было дополнено механизмом авторизации на основе токенов, а доступ к части маршрутов стал возможен только для аутентифицированных пользователей.

## Практическая работа 9

На данном этапе приложение было доработано за счёт реализации механизма refresh-токенов.

### Выполненные доработки

- добавлена генерация `refreshToken`;
- реализован маршрут:

| Маршрут | Метод | Описание |
|---|---|---|
| `/api/auth/refresh` | `POST` | Обновление пары токенов |

### Особенности реализации

Метод `/api/auth/refresh` получает refresh-токен из заголовков запроса и на его основе формирует новую пару токенов:

- `accessToken`
- `refreshToken`

### Формат ответа

После доработки формат ответа при входе пользователя в систему (`POST /api/auth/login`) был приведён к следующему виду:

```json
{
  "accessToken": "",
  "refreshToken": ""
}
